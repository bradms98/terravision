# Terravision Development Notes

## Project Overview

Terravision generates AWS architecture diagrams (draw.io, PNG) from Terraform state and plan files. This is a fork of patrickchugh/terravision with custom enhancements for Sunward's infrastructure.

## Gold Standard Layout

The reference design for diagram output is at `examples/gold-standard-layout.drawio`. The generation script is at `scripts/rearrange_drawio.py`. All renderer changes should be evaluated against this layout.

### Key design principles

- **Resource group containers (S3 buckets, Backup vaults)**: Rendered as a single icon on the left with a text bullet list of child resource type names on the right. NOT a grid of individual icons.
- **Security groups**: Rendered as containers ONLY when inside a VPC/subnet (wrapping EC2 instances, etc.). At the region level, rendered as flat icons with arrows to their resources.
- **Flat region-level icons**: Arranged in a 3-column grid with ~200-230px spacing, max cell width 200px.
- **Service group containers**: 310px wide, height scaled by child count. Arranged in a 3-column grid within the region.
- **Container sizing**: Tight fit-to-content with ~15px gaps between containers.

### Remaining work (from gold standard)

- Section labels (STORAGE, DATA PROTECTION, ANALYTICS, SECURITY, MONITORING & NOTIFICATIONS) not yet auto-generated by the renderer
- Container auto-sizing could be tighter in some cases

## Testing with Docker

### Build the image

```sh
cd /Users/bradengberg/git/terravision
docker build -t terravision:test .
```

### Run against a Scalr-managed workspace

The Docker entrypoint (`docker/entrypoint.sh`) fetches plan JSON and state from Scalr, then runs terravision.

```sh
docker run --rm \
  -e TF_API_TOKEN="$SCALR_TOKEN" \
  -e SCALR_HOSTNAME="sunward.scalr.io" \
  -e WORKSPACE_ID="<workspace-id>" \
  -e DIAGRAM_FILTER="none" \
  -v /path/to/terraform/repo:/workspace/source \
  terravision:test
```

`SCALR_TOKEN` is already set in the shell environment. Use `DIAGRAM_FILTER="network"` for network-only diagrams or `"none"` for all resources.

### Evaluating output

Open the generated `.drawio` file in draw.io and compare to `examples/gold-standard-layout.drawio`. Key things to check:

1. S3/Backup containers use icon + property-list style (not grid of icons)
2. Security groups: containers inside VPCs, flat icons at region level
3. Flat icons in 3-column grid, not horizontally sprawled
4. Diagram width proportional to content (not 4000px+)
5. Containers sized to fit content

## Key Files

| File | Purpose |
|---|---|
| `modules/drawio_renderer.py` | Draw.io XML renderer (layout, sizing, positioning) |
| `modules/drawio_shapes.py` | Draw.io shape/style definitions |
| `modules/graphmaker.py` | Graph construction, resource grouping (S3, Backup) |
| `modules/config/cloud_config_aws.py` | AWS config: GROUP_NODES, ancillary types, relationships |
| `modules/drawing.py` | Region/account hierarchy injection |
| `resource_classes/aws/groups.py` | Group container class definitions |
| `terravision/terravision.py` | Main pipeline orchestration |
| `docker/entrypoint.sh` | Docker entrypoint (Scalr integration) |

## Git Workflow

- Only push to `origin` (bradms98/terravision), never to upstream
- Track changes as GitHub issues in the "Terravision Development" project
- Use plan mode before implementing features
- Always use `.venv` for Python/pip/pytest commands
